# FreeCustom.Email Extension Migration Plan

## 1. Full Architecture

The extension will transition from a heavy client-side, RapidAPI-based mockup into a robust client communicating with the `api2.freecustom.email` backend.

*   **Background Service Worker (`background.js`)**: 
    *   Manages the WebSocket connection to `wss://api2.freecustom.email` (using the extension token).
    *   Acts as the central state manager for caching incoming messages and syncing offline state with the backend.
    *   Listens to WebAuth flow messages to capture and store the `ext_token`.
*   **Popup UI (`src/App.jsx` -> `src/App.tsx` + `email-box`)**: 
    *   Replace the current custom UI with a direct port of the Next.js `email-box.tsx` component to ensure perfect UI/UX parity.
    *   Utilizes Tailwind CSS, Lucide icons, and Shadcn UI components.
    *   State is hydrated via `browser.runtime.sendMessage` communicating with the background worker.
*   **Content Scripts (`content.js`)**: 
    *   Continuously monitors DOM for email and OTP input fields.
    *   Auto-fills the active `freecustom.email` address.
    *   Displays an inline overlay when a new OTP is received via the WebSocket.
*   **Storage (`browser.storage.local`)**:
    *   `ext_token`: The JWT used for authenticating with the backend.
    *   `active_mailbox`: The currently selected email address.
    *   `settings`: User preferences (OTP extraction, notifications, etc.).

## 2. Auth Flow (Extension Token System)

We will implement a secure extension token system to prevent exposing NextAuth session cookies directly to the extension.

1.  **Trigger**: User clicks "Login via FreeCustom.Email" in the extension popup.
2.  **Web Flow**: A new tab opens to `https://www.freecustom.email/ext-auth?source=extension`.
3.  **Frontend validation**: The Next.js app validates the user's NextAuth session. If valid, it requests a token from the backend.
4.  **Backend Issue**: The Node.js backend issues a short-lived `EXT_TOKEN` (JWT) containing `userId`, `plan`, and `fingerprint`.
5.  **Handshake**: The Next.js app securely passes the token to the extension via `window.postMessage` (if injected on a trusted domain) or a redirect to `chrome-extension://[ID]/auth-callback.html`.
6.  **Persistence**: The extension stores the `EXT_TOKEN` and attaches it as `Authorization: Bearer EXT_TOKEN` to all `/ext/*` API requests.

## 3. API Usage Plan

We will create a dedicated `/ext/*` namespace in the backend to explicitly handle extension requests. This isolates extension traffic and allows specific rate-limiting and formatting.

*   **`POST /v1/ext/auth`**: Generate the JWT extension token (called by Next.js frontend).
*   **`GET /v1/ext/mailbox`**: Provision or retrieve the current active mailbox. Generates random guest domains if no token is present.
*   **`GET /v1/ext/messages?mailbox=...`**: Fetch inbox messages. Replaces the local storage simulated folders.
*   **`GET /v1/ext/messages/:id?mailbox=...`**: Fetch full message details, HTML, and attachments.
*   **`DELETE /v1/ext/messages/:id`**: Delete message securely.
*   **`POST /v1/ext/settings`**: Sync cross-tab/cross-browser settings for Free/Pro users.
*   **WebSockets (`wss://api2.freecustom.email/v1/ws`)**: Upgraded to accept `?ext_token=...` in the connection URL or initial payload. Delivers real-time `NEW_MESSAGE` events.
*   **`GET /v1/ext/wait`**: Long-polling fallback API in case WS disconnects or is blocked by strict firewalls.

## 4. Feature Mapping

The backend must validate the `EXT_TOKEN` and enforce pricing tier limitations natively. The extension will map its UI based on the response:

*   **Guest (No Login / No Token)**:
    *   Hits API anonymously; backend rate-limits by IP/Fingerprint.
    *   Can generate a single temp email using basic domains.
    *   No persistence; inbox is wiped if the background script is suspended too long or storage is cleared.
    *   Long-polling or limited WS.
*   **Free (Logged in via Token, `plan=free`)**:
    *   Inbox sync across multiple tabs/devices using the token.
    *   Limited real-time WebSockets.
    *   Access to standard shared domains.
    *   Auto OTP detection enabled.
*   **Pro (Logged in via Token, `plan=pro`)**:
    *   Instant, persistent WebSockets.
    *   Access to premium domains (`@pro-domain.com`).
    *   Multiple concurrent active inboxes managed via the extension UI.
    *   Priority background monitoring.

## 5. Migration Plan (QuickTemp → FreeCustom.Email)

**Phase 1: Extension Preparation**
1.  Update `manifest.json` (Name: "FreeCustom.Email – Temp Mail & OTP Inbox", SEO description).
2.  Clean up `background.js`: Remove all RapidAPI keys, local simulated spam filters, and hardcoded domains.

**Phase 2: Backend & Frontend API Scaffolding**
1.  **Backend (`fce-backend`)**: Create `api/src/v1/routes/ext.ts`. Add the `ext_token` generation logic in `api-auth.ts`.
2.  **Frontend (`fce-frontend`)**: Create the `/ext-auth` route in Next.js to facilitate the login handshake.

**Phase 3: State & Data Layer Migration**
1.  Implement API utility (`api.js` or `api.ts`) in the extension to automatically attach the `EXT_TOKEN`.
2.  Wire up the new WebSockets logic in `background.js` to point to the production backend.
3.  Ensure `browser.storage.local` is only used for caching, not as the source of truth for email states (Inbox/Spam/Trash).

**Phase 4: UI/UX Synchronization**
1.  Copy the Next.js `email-box` React components and required UI components (Tailwind/Shadcn) into the extension `src/components` folder.
2.  Adapt the hooks and local states in `email-box.tsx` to fetch from the extension's background worker instead of Next.js `/api` routes.

**Phase 5: Testing & Edge Cases**
1.  Handle token expiration (auto-refresh or prompt login).
2.  Ensure service worker wake-up events correctly reconnect the WebSocket.
3.  Test OTP auto-fill `content.js` with the new data structures coming from the backend.
